# 文档规范详解 — product-doc-lifecycle v3.1

## 快速参考：项目复杂度评估

在建立文档体系前，先评估项目复杂度，只建必要文档。

### 复杂度评估表（逐项打分）

| 评估维度 | 简单（0分） | 中等（1分） | 复杂（2分） |
|---------|--------------|-------------|-------------|
| 团队规模 | 1-3人 | 3-8人 | 8+人 |
| 架构形态 | 单应用 / 无微服务 | 前后端分离 / 1-2子项目 | 微服务 / 3+子项目 |
| 代码仓库 | 单仓库 | 2个仓库 | 多仓库 / monorepo |
| 项目周期 | <3个月 | 3-12个月 | >12个月 |
| 部署形态 | 单节点 / 无专职运维 | 前后端分部署 | 多节点 / 容器化 / 专职DevOps |
| 外部依赖 | 很少（≤3） | 中等（4-10） | 很多（10+） |

**总分 → 文档策略**：
- 0-3分：**简单** → 最小文档集
- 4-7分：**中等** → 标准文档集
- 8-12分：**复杂** → 完整文档集

### 01-requirements — 需求规格说明书

**职责**：定义"做什么"，不涉及"怎么做"。

**必须包含**：
- 项目背景与目标
- 核心功能列表（功能模块 → 用户故事）
- 非功能性需求（性能、安全、并发）
- 验收标准（每条需求可测试）
- 范围边界（明确不做什么）

**最小模板结构**：
```markdown
# 需求规格说明书

> **版本**: V{ver} | **状态**: {草稿/评审中/已冻结}

## 1. 项目背景
## 2. 技术目标（可量化指标）
## 3. 功能需求
   ### 3.1 {模块名}
   - 功能描述
   - 验收标准
## 4. 非功能性需求
## 5. 范围边界（Out of Scope）

**最后更新**: {日期} | **状态**: {草稿/评审中/已冻结}
```

### 02-architecture — 系统架构设计

**职责**：定义系统的骨架——组件、关系、边界、技术选型。

**必须包含**：
- 系统组件图（ASCII 或 Mermaid）
- 服务分层说明（每层职责）
- 端口规划表（端口 / 服务 / 用途）
- 技术栈一览表
- 数据流说明（请求如何流经各服务）
- 关键设计决策记录（ADR）

**最小模板结构**：
```markdown
# 系统架构设计

> **版本**: V{ver}

## 1. 架构总览（组件图）
## 2. 服务分层
## 3. 端口规划
| 端口 | 服务 | 用途 |
## 4. 技术栈
| 分类 | 技术 | 版本 |
## 5. 数据流
## 6. 设计决策记录（ADR）

**最后更新**: {日期}
```

### 03-database — 数据库设计

**职责**：数据模型的权威来源。

**必须包含**：
- 数据库类型与版本
- 所有表结构（表名 / 字段 / 类型 / 约束 / 注释）
- 索引设计
- 表间关系（ER 图描述）
- 迁移历史（每次 schema 变更记录）

**最小模板结构**：
```markdown
# 数据库设计 V{ver}

## 1. 数据库规范
## 2. 核心表
   ### {table_name}
   | 字段 | 类型 | 约束 | 说明 |
## 3. 索引
## 4. 表关系
## 5. 迁移记录
| 版本 | 变更 | 日期 |

**最后更新**: {日期}
```

### 04-api — API 接口文档

**职责**：所有对外接口的权威来源。

**必须包含**：
- 基础 URL
- 认证方式（JWT / API Key 等）
- 每个端点：方法 / 路径 / 请求参数 / 响应格式 / 错误码
- 通用错误码表
- 与 `docs/api/openapi.yaml` 规范文件的对应关系说明

**OpenAPI 规范文件（推荐）**：

项目应维护一份机器可读的 OpenAPI/Swagger 规范文件 `docs/api/openapi.yaml`，作为 API 契约的权威机器表示。`04-api.md` 为人读版本，`openapi.yaml` 为机器可读版本，两者**必须保持同步**。

| 场景 | 基于 openapi.yaml 自动化 |
|------|----------------------|
| 前端 Mock 数据 | 通过 MSW / json-server 自动生成 |
| SDK 代码生成 | 使用 openapi-generator 生成客户端 |
| API 自动化测试 | 基于规范文件生成测试用例 |
| 接口文档托管 | 导入 Swagger UI / Redoc 展示 |
| API 变更检测 | CI 中 diff 比对规范文件变更 |

**OpenAPI 文件规范**：
- 放置路径：`docs/api/openapi.yaml`
- 格式：OpenAPI 3.0+ 规范（YAML 格式）
- 与 `04-api.md` 同步更新——修改任一方时必须同步另一方
- 版本号与核心文档版本号一致

**最小模板结构**：
```markdown
# API 接口文档

> **版本**: V{ver} | **最后更新**: {日期}

## 1. 基础信息
- Base URL: http://...
- 认证: Bearer Token (JWT)
- OpenAPI 规范: [openapi.yaml](../api/openapi.yaml)

## 2. 接口列表
   ### {模块名}
   #### POST /path
   **描述**: ...
   **请求体**: `{ "field": "type" }`
   **响应**: `{ "code": 200, "data": {} }`

## 3. 错误码

**最后更新**: {日期}
```

### 05-feature-guide — 功能指南

**职责**：解释"如何使用"各功能模块，面向开发者和测试人员。

**必须包含**：
- 每个功能模块的使用说明
- 配置项说明
- 常见使用场景示例
- 常见问题（FAQ）

---

### 06-deployment — 部署与运维手册

**职责**：让任何人能独立部署和运维系统。

**必须包含**：
- 环境要求（OS / 内存 / 磁盘 / 依赖）
- 安装步骤（顺序明确）
- 配置项说明（环境变量 / 配置文件）
- 常用运维命令
- 健康检查方法
- 故障排查手册（症状 → 原因 → 解决方案）
- 备份与恢复流程

---

### 07-release-notes — 版本发布说明

**职责**：每次发布的摘要，面向所有角色（含非技术）。

**必须包含**：
- 版本号 + 发布日期
- 新增功能摘要
- Bug 修复摘要
- 破坏性变更警告（Breaking Changes）
- 升级指引（从上一版本升级的操作）

---

### 08-user-guide — 产品使用说明

**职责**：面向**最终用户**（非开发者），教用户"怎么用这个产品"，是产品交付时的用户手册。

> **与 05-feature-guide 的区别**：
> - 05 面向开发者/测试人员，描述功能实现和使用（含配置项、技术细节）
> - 08 面向最终用户，只描述用户操作和预期效果（不含技术细节）

**何时创建**：
- 项目有终端用户界面（Web UI / 移动端 / 桌面端）→ 必须建
- 项目是纯 API / 后台服务 / 库 → 不建（用 04-api + README 代替）
- B2C / B2B SaaS → 交付前必须完成
- 内部工具 → 建议建（减少培训成本）

**必须包含**：
- 产品简介（是什么、能做什么、适用人群）
- 快速开始（首次使用的最小步骤）
- 各功能模块的使用说明（面向用户语言，非技术语言）
- 常见操作流程（步骤列表，可配截图/图示）
- 常见问题（FAQ）
- 联系方式（反馈渠道、版本更新说明链接）

**不写**：
- 技术实现细节（属于 02-architecture）
- API 参数说明（属于 04-api）
- 配置项/环境变量（属于 05-feature-guide 或 06-deployment）
- 代码/命令行操作（最终用户不需要）

**最小模板结构**：
```markdown
# {产品名} 使用说明

> **版本**: V{ver} | **最后更新**: {日期} | **适用产品版本**: V{ver}

## 1. 产品简介
```

## 二、版本号同步规则

### 核心文档版本号

发布 `V{X}.{Y}.{Z}` 时，以下位置必须同步：

| 位置 | 更新内容 |
|------|---------|
| `docs/core/*.md` 文件头部版本号字段 | 更新为 `V{new}` |
| `docs/api/openapi.yaml`（如存在） | `info.version` 更新为 `{X}.{Y}.{Z}` |
| `CHANGELOG.md` | 新增版本条目 |
| `README.md` 版本历史表 | 新增一行 |
| `docs/DOCUMENTATION_INDEX.md` | 更新所有受影响文档的版本号 |
| `07-release-notes` | 新建或追加版本发布章节 |

### 子项目文档版本号

- 子项目维护**独立版本号**，不强制与核心版本对齐
- 子项目版本号随子项目自身发布节奏递增
- 子项目版本号必须记录在 `DOCUMENTATION_INDEX.md` 的子项目表格中

**快速验证命令（PowerShell）**：
```powershell
# 检查 docs/core 目录下版本号是否一致
Get-ChildItem docs/core/*.md | ForEach-Object {
    $_ | Select-String -Pattern "V\d+\.\d+\.\d+" | Select-Object -First 1
}
```

## 三、AI Agent 协作规则

### Agent 读取顺序（上下文加载优先级）

1. `README.md` — 了解项目全貌
2. `02-architecture_V*.md` — 理解系统结构
3. `04-api_V*.md` — 理解接口契约
4. `03-database_V*.md` — 理解数据模型
5. 子项目文档按需加载
6. 其他按需加载

### Agent 写文档规则

- 修改代码后，**立即**在同一 turn 更新对应文档
- 更新文档时，**先读后写**（Read → Edit，禁止直接覆写）
- 文档中的版本号、日期字段**必须**更新
- 工作记忆（`.workbuddy/memory/`）记录进度，**不替代**正式文档
- 子项目操作时，使用**作用域编号**命名（`{subproject}-{NN}-{type}.md`）

### Agent 检查文档完整性（项目初次接手时）

```
1. 列出 docs/ 目录结构
2. 对照核心文档清单，找出缺失文档
3. 列出所有子项目目录，检查每个子项目的文档完整性
4. 向用户报告：
   - 核心文档：已有 {N}/9 份必需文档，缺少：{列表}
   - 子项目文档：{子项目名} 已有 {M}/7 份，缺少：{列表}
5. 询问是否立即创建缺失文档
```

## 四、子项目文档内容边界详解

每个子项目的文档集采用**作用域编号**体系，文件名包含子项目 ID 前缀，与核心文档编号完全解耦。

### 命名格式

```
{子项目ID}-{序号}-{英文名}.md
```

示例：`admin-01-requirements.md`、`ai-services-04-api.md`

### 文档类型固定映射

| 序号 | 英文名 | 类型 | 必须？ |
|------|--------|------|--------|
| 01 | `requirements` | 功能需求 | ✅ 必须 |
| 02 | `architecture` | 功能架构设计 | ✅ 必须 |
| 03 | `data-model` | 数据模型 | 按需 |
| 04 | `api` | API / 接口 | 按需 |
| 05 | `feature-guide` | 功能使用指南 | 按需 |
| 06 | `ui-design` | UI/交互设计 | 纯前端 |
| 07 | `deployment` | 部署说明 | 独立部署 |

> **同类文档在所有子项目中编号一致**——对比 admin 和 ai-services 的架构？直接看 `*-02-architecture`。

### {subproject}-01-requirements — 子项目功能需求

**职责**：定义该子项目"做什么"，聚焦子项目自身边界。

**必须包含**：
- 子项目目标与定位（在整体系统中的角色）

- 用户故事（面向该子项目的使用者）
- 验收标准
- 与其他子项目的依赖关系说明（依赖哪些接口/服务）

**不写**：整体系统需求（核心 `01-requirements` 负责）

### {subproject}-02-architecture — 子项目功能架构设计

**职责**：描述子项目内部的技术结构，不涉及整体系统架构。

**必须包含**：
- 子项目内部模块/组件划分图（Mermaid 或 ASCII）
- 各模块职责说明
- 技术选型（本子项目使用的框架/库）
- 内部数据流（模块间如何传递数据）
- 与外部系统/服务的集成点（调用哪些外部 API，依赖哪些服务）
- 关键设计决策（如状态管理方案、缓存策略）

**前端子项目还需包含**：
- 路由结构（页面路径树）
- 组件树（主要组件层级）
- 状态管理架构（Store 设计）

**后端/微服务子项目还需包含**：
- 服务内部分层（Controller / Service / Repository）
- 消息队列/事件设计（如有）
- 并发/限流设计
- **微服务专属章节**（如为微服务子项目，见 SKILL.md 6.9 节）：
  - 服务发现与注册
  - 服务间通信（同步/异步）
  - 数据一致性策略（分布式事务）
  - 容错与降级（熔断/限流/降级）
  - 服务依赖拓扑

**不写**：整体系统组件图（核心 `02-architecture` 负责）

**最小模板结构**：
```markdown
# {子项目名} 功能架构设计

## 1. 子项目定位
（在整体系统中的角色，一段话）

## 2. 内部模块划分
（组件/模块图）

## 3. 技术栈
| 分类 | 技术 | 版本 | 选型原因 |

## 4. 内部数据流

## 5. 外部依赖
| 依赖服务 | 调用方式 | 用途 |

## 6. 设计决策记录

**最后更新**: {日期} | **版本**: V{ver}
```

### {subproject}-03-data-model — 子项目数据模型

**职责**：子项目专属的数据结构、表或状态定义。

**适用场景**：子项目拥有独立数据库、专属表，或复杂的前端状态模型时创建。

**必须包含**（后端/数据库类子项目）：
- 子项目专属的表结构（字段/类型/约束）
- 与核心数据库的关联关系（外键/引用）
- 专属索引设计

**必须包含**（前端子项目）：
- Pinia/Vuex Store 结构
- 主要数据对象的 TypeScript 类型定义
- 本地缓存/持久化策略

**不写**：整体数据库所有表（核心 `03-database` 负责）

### {subproject}-04-api — 子项目 API 文档

**职责**：子项目对外暴露的接口，或子项目消费外部接口的封装规范。

**后端/微服务子项目（对外暴露接口）**：
- 该子项目暴露的所有端点（方法/路径/请求/响应）
- 认证方式（若与全局不同）
- 子项目专属错误码
- 与核心 `04-api` 的对应关系（引用而非重复）

**前端子项目（消费接口）**：
- API 客户端封装规范（如 `useUserApi()` 的使用方式）
- 请求/响应拦截约定
- 错误处理统一规范
- Mock 数据策略（开发阶段）
- 调用的后端接口清单（引用核心 `04-api` 的链接）

**不写**：所有接口的权威定义（核心 `04-api` 负责）

## 六、跨文档引用规范

### 引用格式

所有跨文档引用使用**相对路径** + **版本号** + **锚点**：

```markdown
<!-- 子项目 → 核心文档 -->
详见 [核心 API §认证接口](../core/04-api.md#认证接口)

<!-- 核心 → 子项目文档 -->
详见 [admin 子项目 API 规范](admin/admin-04-api.md)

<!-- 子项目 → 子项目 -->
详见 [ai-services 数据模型](../ai-services/ai-services-03-data-model.md)

<!-- 引用具体章节 -->
[核心架构 §端口规划](../core/02-architecture.md#端口规划)
```

### 引用规则

1. **使用相对路径**（`../`），不使用绝对路径
2. **文件名必须包含版本号**，确保引用精确
3. **锚点指向具体章节标题**，避免引用整篇文档
4. **引用而非复制**——需要外部信息时创建引用链接，不复制内容
5. **更新联动**——被引用文档变更时，检查并更新引用链接

## 七、子项目生命周期规范

子项目从创建到废弃经历五个阶段，详见 SKILL.md §子项目生命周期。此处补充各阶段的文档检查清单。

### 创建阶段检查清单

```
- [ ] 子项目 ID 已确定（kebab-case，2-20 字符）
- [ ] 子项目目录已创建于 docs/{subproject}/
- [ ] DOCUMENTATION_INDEX.md 已注册子项目
- [ ] {subproject}-01-requirements 已创建（状态: 草稿）
- [ ] {subproject}-02-architecture 已创建（状态: 草稿）
- [ ] 核心 02-architecture 已更新（添加子项目节点）
```

### 冻结阶段检查清单

```
- [ ] 所有文档状态已改为"已冻结"
- [ ] 无遗留 TODO 或占位内容
- [ ] DOCUMENTATION_INDEX.md 状态已更新
- [ ] 所有文档版本号已更新为冻结版本
```

### 归档阶段检查清单

```
- [ ] 子项目目录已移至 docs/_archived/{subproject}/
- [ ] DOCUMENTATION_INDEX.md 已更新（移至归档表格）
- [ ] 核心文档中的引用已标注"（已归档）"
- [ ] 团队已通知
```

### 废弃阶段检查清单

- [ ] 确认子项目彻底下线，不再需要任何参考
- [ ] 保留 `docs/_archived/{subproject}/` 目录（不删除）
- [ ] 在每份文档头部添加：`> ⚠️ **已废弃** — 该子项目已于 {日期} 正式废弃，文档仅供参考。`
- [ ] `DOCUMENTATION_INDEX.md` 中标注"已废弃"
- [ ] 通知相关团队成员

### 生命周期状态流转规则

| 当前状态 | 可流转至 | 前置条件 |
|---------|---------|---------|
| 草稿 | 开发中 | 需求评审通过 |
| 开发中 | 已冻结 | 所有文档完整、功能验收通过 |
| 已冻结 | 归档 | 子项目不再需要活跃维护 |
| 已冻结 | 开发中 | 需要恢复开发（罕见，需审批） |
| 归档 | 已废弃 | 确认彻底下线 |

**禁止**：直接从"开发中"跳到"归档"或"废弃"，必须先经过"冻结"阶段。
