## 二、作品概述

### 2.1 一句话简介

Product-Doc-Lifecycle 是一个为 AI Agent（WorkBuddy）设计的 Skill，能够根据项目复杂度和当前开发阶段，**自动决策"该建什么文档、放哪里、什么时候更新"**，彻底解决软件项目中文档缺失或过度文档化的痛点。

### 2.2 作品定位

本作品是一个 **AI Agent 技能插件（Skill）**，运行在 WorkBuddy AI 助手中，面向软件开发团队，提供智能化的文档管理体系。

### 2.3 核心价值

- 🎯 **按需建文档**：根据项目复杂度（简单/中等/复杂）自动匹配文档集，拒绝一刀切
- 📅 **分阶段推进**：按开发阶段（S0-S6）逐步建立文档，防止"前期建一堆空文档没人维护"
- 🤖 **AI 驱动**：自然语言触发，开发者只需说"帮我初始化文档体系"，AI 自动完成评估和执行
- 📋 **规范内置**：内置文档规范、模板和检查清单，确保文档质量

## 三、问题背景与需求分析

### 3.1 痛点描述

在软件开发过程中，文档管理一直是个难题：

| 问题 | 具体表现 | 后果 |
|------|---------|------|
| **文档缺失** | 项目交付时找不到架构图、API文档 | 交接困难，维护成本高 |
| **过度文档化** | 小项目照搬大厂规范，建立几十份文档 | 维护负担重，文档腐烂 |
| **时机不对** | 初期建好所有文档，后期无人更新 | 文档与代码脱节 |
| **无标准规范** | 每人写文档风格不同，质量参差不齐 | 可读性差，团队协作受阻 |

### 3.2 目标用户

- 软件开发团队（1-20人）
- 使用 AI Agent 辅助开发的个人开发者
- 需要规范化文档管理的开源项目维护者

### 3.3 市场需求

随着 AI 编程助手的普及，开发者越来越依赖 AI 进行代码和文档管理。但现有 AI 工具缺乏**对项目上下文的理解**，无法智能决策文档范围。本作品填补了这一空白。

## 四、技术方案

### 4.1 整体架构

```
用户（自然语言）
    ↓
WorkBuddy AI Agent
    ↓
product-doc-lifecycle Skill
    ├── 复杂度评估引擎（自动评估项目规模）
    ├── 阶段判断引擎（识别当前开发阶段）
    ├── 文档决策引擎（决策树：该建什么文档）
    └── 文档生成引擎（基于模板自动生成）
    ↓
输出：结构化文档集（Markdown格式）
```

### 4.2 核心模块

#### Module 1：复杂度评估引擎

通过三个维度评估项目复杂度：

| 维度 | 简单 | 中等 | 复杂 |
|------|------|------|------|
| 团队规模 | 1-3人 | 3-8人 | 8+人 |
| 架构形态 | 单应用 | 前后端分离 | 微服务/多仓库 |
| 项目周期 | <3个月 | 3-12个月 | >12个月 |

**评分规则**：总分 0-3 → 简单，4-7 → 中等，8-12 → 复杂

#### Module 2：阶段判断引擎

| 阶段 | 触发条件 | 建立文档 |
|------|---------|---------|
| S0 初始化 | 仓库创建 | README + CHANGELOG |
| S1 需求 | 需求评审通过 | +01-requirements |
| S2 架构 | 技术方案确定 | +02-architecture |
| S3 接口 | API 第一稿完成 | +04-api |
| S4 开发 | 首个功能完成 | +05-feature-guide |
| S5 部署 | 首次部署 | +06-deployment +08-user-guide |
| S6 发布 | 首次发布 | +07-release-notes |

#### Module 3：文档决策树

```
项目复杂度？
├── 简单（1-3人）→ 4份核心文档
├── 中等（3-8人）→ 7份文档
└── 复杂（8+人）→ 完整8份 + 子项目文档
```

#### Module 4：文档生成引擎

基于内置模板（templates.md），自动生成符合规范的 Markdown 文档，包括：
- 固定格式头（项目名称、版本、更新日期）
- 标准化章节结构
- 检查清单（Checklist）

### 4.3 技术实现

| 技术点 | 实现方式 |
|--------|---------|
| Skill 定义 | SKILL.md（YAML frontmatter + Markdown指令）|
| 文档模板 | references/templates.md（可复制的完整模板）|
| 文档规范 | references/doc-spec.md（内容边界、质量要求）|
| 触发机制 | 自然语言匹配（"帮我初始化文档体系"）|
| 输出格式 | Markdown 文件，兼容 Git 版本管理 |

## 六、创新点

### 6.1 核心创新

> **将"文档决策"这一原本依赖人工经验的过程，转化为可执行的 AI Skill**

1. **动态文档策略**：首次提出"按复杂度和阶段动态调整文档范围"的理念，打破"一刀切"的文档规范
2. **AI 原生设计**：专为 AI Agent 设计，通过自然语言触发，而非传统命令行工具
3. **防腐烂机制**：分阶段建立文档，确保每份文档在建立时都有实际内容支撑

### 6.2 与现有方案对比

| 对比维度 | 传统文档工具（Notion/Confluence） | 静态文档模板（Readme模板） | **本作品** |
|---------|---------------------------|---------------------|---------|
| 智能决策 | ❌ 手动选择 | ❌ 固定模板 | ✅ AI自动决策 |
| 动态调整 | ❌ 不支持 | ❌ 不支持 | ✅ 按阶段推进 |
| 项目适配 | ❌ 一刀切 | ❌ 无适配 | ✅ 按复杂度适配 |
| 维护提醒 | ⚠️ 需手动设置 | ❌ 无 | ✅ AI主动提醒 |

## 七、应用场景

### 场景1：初创项目快速启动

**用户**：独立开发者，启动新 Side Project  
**问题**：不知道该建哪些文档，建多了维护不过来，建少了后期后悔  
**方案**：对 AI 说"帮我初始化文档体系"，AI 自动评估为"简单项目"，建立4份核心文档

### 场景2：团队协作规范化

**用户**：5人创业团队，走出初创期  
**问题**：团队成员文档风格不一，新成员入职后不知道文档在哪里  
**方案**：AI 评估为"中等复杂度"，建立7份规范文档，并生成统一模板

### 场景3：微服务架构管理

**用户**：大型企业项目，20+微服务  
**问题**：各服务文档各自为政，缺乏统一规范  
**方案**：AI 评估为"复杂项目"，建立完整文档集 + 为每个子服务生成独立文档

## 八、使用说明

### 8.1 安装方式

**方式一：本地安装**
```bash
# 将 product-doc-lifecycle/ 复制到 WorkBuddy skills 目录
cp -r product-doc-lifecycle ~/.workbuddy/skills/
```

**方式二：Skill Hub 安装（待发布）**
```
/workbuddy install product-doc-lifecycle
```

### 8.2 快速上手

安装后，直接对 AI 说：

| 你说的话 | AI 做什么 |
|---------|---------|
| "帮我初始化文档体系" | 评估复杂度，建立最小必要文档集 |
| "检查文档完整性" | 对照当前阶段，报告缺失文档 |
| "这个项目该写什么文档？" | 根据复杂度和阶段给出建议 |
| "建立子项目文档" | 为子项目生成独立文档集 |

### 8.3 文档输出示例

执行"帮我初始化文档体系"后，项目目录结构：

```
your-project/
├── README.md              ← 自动生成
├── CHANGELOG.md          ← 自动生成
├── docs/
│   ├── 01-requirements.md   ← 需求规格（按需）
│   ├── 02-architecture.md   ← 系统架构（中等及以上）
│   ├── 04-api.md            ← API接口文档（始终）
│   └── 08-user-guide.md     ← 用户使用说明（中等及以上）
```
