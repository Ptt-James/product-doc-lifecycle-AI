# 文档规范详解 — documentation-lifecycle v3.1

> SKILL.md 的规范详解，专注于各文档的**内容边界、检查清单、版本同步规则**。

---

## 一、复杂度评估评分表

在建立文档体系前，用此表评估项目复杂度，决定最小文档集。

| 评估维度 | 简单（0分） | 中等（1分） | 复杂（2分） |
|---------|------------|------------|------------|
| 团队规模 | 1-3人 | 3-8人 | 8+人 |
| 架构形态 | 单应用 / 无微服务 | 前后端分离 / 1-2子项目 | 微服务 / 3+子项目 |
| 代码仓库 | 单仓库 | 2个仓库 | 多仓库 / monorepo |
| 项目周期 | <3个月 | 3-12个月 | >12个月 |
| 部署形态 | 单节点 / 无专职运维 | 前后端分部署 | 多节点 / 容器化 / 专职DevOps |
| 外部依赖 | 很少（≤3） | 中等（4-10） | 很多（10+） |

**总分 → 文档策略**：0-3分 简单 / 4-7分 中等 / 8-12分 复杂

## 三、各核心文档内容边界

---

### 01-requirements — 需求规格说明书

**职责**：定义"做什么"，不涉及"怎么做"。

**必须包含**：
- 项目背景与目标
- 核心功能列表（功能模块 → 用户故事）
- 非功能性需求（性能、安全、并发）
- 验收标准（每条需求可测试）
- 范围边界（明确不做什么）

---

### 02-architecture — 系统架构设计

**职责**：定义系统的骨架——组件、关系、边界、技术选型。

**必须包含**：
- 系统组件图（ASCII 或 Mermaid）
- 服务分层说明（每层职责）
- 端口规划表（端口 / 服务 / 用途）
- 技术栈一览表
- 数据流说明（请求如何流经各服务）
- 关键设计决策记录（ADR）

---

### 03-database — 数据库设计

**职责**：数据模型的权威来源。

**必须包含**：
- 数据库类型与版本
- 所有表结构（表名 / 字段 / 类型 / 约束 / 注释）
- 索引设计
- 表间关系（ER 图描述）
- 迁移历史（每次 schema 变更记录）

---

### 04-api — API 接口文档

**职责**：所有对外接口的权威来源。

**必须包含**：
- 基础 URL 与认证方式（JWT / API Key 等）
- 每个端点：方法 / 路径 / 请求参数 / 响应格式 / 错误码
- 通用错误码表
- 与 `docs/api/openapi.yaml` 的对应关系说明

**OpenAPI 规范文件（推荐）**：

项目应维护 `docs/api/openapi.yaml`（机器可读）与 `04-api.md`（人读）两份，**必须保持同步**。

| 场景 | 基于 openapi.yaml 自动化 |
|------|----------------------|
| 前端 Mock 数据 | MSW / json-server 自动生成 |
| SDK 代码生成 | openapi-generator 生成客户端 |
| API 自动化测试 | 基于规范文件生成测试用例 |
| 接口文档托管 | 导入 Swagger UI / Redoc 展示 |
| API 变更检测 | CI 中 diff 比对规范文件变更 |

---

### 05-feature-guide — 功能指南

**职责**：解释"如何使用"各功能模块，面向开发者和测试人员。

**必须包含**：
- 每个功能模块的使用说明
- 配置项说明
- 常见使用场景示例
- 常见问题（FAQ）

---

### 06-deployment — 部署与运维手册

**职责**：让任何人能独立部署和运维系统。

**必须包含**：
- 环境要求（OS / 内存 / 磁盘 / 依赖）
- 安装步骤（顺序明确）
- 配置项说明（环境变量 / 配置文件）
- 常用运维命令
- 健康检查方法
- 故障排查手册（症状 → 原因 → 解决方案）
- 备份与恢复流程

---

### 07-release-notes — 版本发布说明

**职责**：每次发布的摘要，面向所有角色（含非技术）。

**必须包含**：
- 版本号 + 发布日期
- 新增功能摘要
- Bug 修复摘要
- 破坏性变更警告（Breaking Changes）
- 升级指引

---

### 08-user-guide — 产品使用说明

**职责**：面向**最终用户**（非开发者），教用户"怎么用这个产品"。

> **与 05-feature-guide 的区别**：
> - 05 面向开发者/测试人员，含配置项、技术细节
> - 08 面向最终用户，只描述用户操作和预期效果

**何时创建**：
- 有终端用户界面（Web UI / 移动端 / 桌面端）→ 必须建
- 纯 API / 后台服务 / 库 → 不建（用 04-api + README 代替）
- B2C / B2B SaaS → 交付前必须完成

**必须包含**：产品简介 / 快速开始 / 各功能使用说明（用户语言）/ 常见操作流程 / FAQ / 联系方式

**不写**：技术实现细节、API 参数、配置项/环境变量、代码/命令行操作

---

### CHANGELOG.md — 变更日志

**格式遵循 [Keep a Changelog](https://keepachangelog.com/) + Conventional Commits**：

```markdown
## [Unreleased]

## [1.2.0] - 2026-04-30
### Added
### Fixed
### Changed
### Breaking Changes
```

**强制规则**：
- 每次发布前，把 `[Unreleased]` 内容移至新版本号
- 版本号与代码 tag 严格同步
- 禁止在已发布版本条目中追溯修改

---

### docs/DOCUMENTATION_INDEX.md — 文档索引

**职责**：所有文档的导航入口，保持版本号与实际文件同步。模板见 templates.md。

## 四、版本号同步规则

发布 `V{X}.{Y}.{Z}` 时，以下位置必须同步：

| 位置 | 更新内容 |
|------|---------|
| `docs/core/*.md` 文件头部版本号字段 | 更新为 `V{new}` |
| `docs/api/openapi.yaml`（如存在） | `info.version` 更新 |
| `CHANGELOG.md` | 新增版本条目 |
| `README.md` 版本历史表 | 新增一行 |
| `docs/DOCUMENTATION_INDEX.md` | 更新所有受影响文档的版本号 |
| `07-release-notes` | 新建或追加版本发布章节 |

**子项目文档版本号**：
- 子项目维护**独立版本号**，不强制与核心版本对齐
- 版本号必须记录在 `DOCUMENTATION_INDEX.md` 的子项目表格中

**快速验证命令（PowerShell）**：
```powershell
Get-ChildItem docs/core/*.md | ForEach-Object {
    $_ | Select-String -Pattern "V\d+\.\d+\.\d+" | Select-Object -First 1
}
```

## 五、子项目文档内容边界

每个子项目的文档集采用**作用域编号**体系（见 SKILL.md §三 命名规范）。

### 文档类型固定映射

| 序号 | 英文名 | 类型 | 必须？ |
|------|--------|------|--------|
| 01 | `requirements` | 功能需求 | 必须 |
| 02 | `architecture` | 功能架构设计 | 必须 |
| 03 | `data-model` | 数据模型 | 按需 |
| 04 | `api` | API / 接口 | 按需 |
| 05 | `feature-guide` | 功能使用指南 | 按需 |
| 06 | `ui-design` | UI/交互设计 | 纯前端 |
| 07 | `deployment` | 部署说明 | 独立部署 |

> 同类文档在所有子项目中编号一致——对比 admin 和 ai-services 的架构？直接看 `*-02-architecture`。

---

### {subproject}-01-requirements — 子项目功能需求

**职责**：定义该子项目"做什么"，聚焦子项目自身边界。

**必须包含**：子项目目标与定位 / 功能模块列表 / 用户故事 / 验收标准 / 与其他子项目的依赖关系

**不写**：整体系统需求（核心 `01-requirements` 负责）

---

### {subproject}-02-architecture — 子项目功能架构设计

**职责**：描述子项目内部的技术结构，不涉及整体系统架构。

**必须包含**：
- 子项目内部模块/组件划分图
- 各模块职责说明
- 技术选型（本子项目使用的框架/库）
- 内部数据流
- 与外部系统/服务的集成点
- 关键设计决策

**前端子项目还需包含**：路由结构 / 组件树 / 状态管理架构

**后端/微服务子项目还需包含**：
- 服务内部分层（Controller / Service / Repository）
- 消息队列/事件设计（如有）
- 并发/限流设计
- **微服务专属章节**（见下方）

**不写**：整体系统组件图（核心 `02-architecture` 负责）

### {subproject}-03-data-model — 子项目数据模型

**适用场景**：子项目拥有独立数据库、专属表，或复杂的前端状态模型时创建。

**后端类**：子项目专属表结构 / 与核心数据库的关联 / 专属索引

**前端类**：Pinia/Vuex Store 结构 / 主要 TypeScript 类型定义 / 本地缓存策略

**不写**：整体数据库所有表（核心 `03-database` 负责）

---

### {subproject}-04-api — 子项目 API 文档

**后端子项目（对外暴露接口）**：该子项目暴露的所有端点 / 认证方式 / 子项目专属错误码 / 与核心 `04-api` 的对应关系（引用而非重复）

**前端子项目（消费接口）**：API 客户端封装规范 / 请求/响应拦截约定 / 错误处理统一规范 / Mock 数据策略 / 调用的后端接口清单（引用核心 `04-api` 链接）

**不写**：所有接口的权威定义（核心 `04-api` 负责）

---

### {subproject}-05-feature-guide — 子项目功能指南

各功能页面/模块的使用说明 / 关键配置项 / 常见操作流程 / FAQ

**不写**：底层实现代码细节

---

### {subproject}-06-ui-design — 子项目 UI/交互设计（纯前端专用）

页面列表与路由路径 / 页面布局描述 / 关键交互逻辑 / 色彩/字体/组件规范引用

**不写**：API 接口细节、数据库字段

---

### {subproject}-07-deployment — 子项目部署说明

子项目专属环境要求 / 独立构建与部署步骤 / 子项目专属配置项 / 健康检查端点 / 与核心 `06-deployment` 的关系（引用）

**不写**：全局基础设施配置（核心 `06-deployment` 负责）

## 八、子项目生命周期检查清单

> 生命周期阶段定义和状态流转规则见 SKILL.md §六。

### 创建阶段

```
- [ ] 子项目 ID 已确定（kebab-case，2-20 字符）
- [ ] 子项目目录已创建于 docs/{subproject}/
- [ ] DOCUMENTATION_INDEX.md 已注册子项目
- [ ] {subproject}-01-requirements 已创建（状态: 草稿）
- [ ] {subproject}-02-architecture 已创建（状态: 草稿）
- [ ] 核心 02-architecture 已更新（添加子项目节点）
```

### 冻结阶段

```
- [ ] 所有文档状态已改为"已冻结"
- [ ] 无遗留 TODO 或占位内容
- [ ] DOCUMENTATION_INDEX.md 状态已更新
- [ ] 所有文档版本号已更新为冻结版本
```

### 归档阶段

```
- [ ] 子项目目录已移至 docs/_archived/{subproject}/
- [ ] DOCUMENTATION_INDEX.md 已更新（移至归档表格）
- [ ] 核心文档中的引用已标注"（已归档）"
- [ ] 团队已通知
```

### 废弃阶段

- [ ] 确认子项目彻底下线，不再需要任何参考
- [ ] 保留 `docs/_archived/{subproject}/` 目录（不删除）
- [ ] 在每份文档头部添加：`> ⚠️ **已废弃** — 该子项目已于 {日期} 正式废弃，文档仅供参考。`
- [ ] `DOCUMENTATION_INDEX.md` 中标注"已废弃"
- [ ] 通知相关团队成员
